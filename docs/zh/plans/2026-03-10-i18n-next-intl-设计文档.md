# Mission Control i18n（next-intl）设计文档

> 本文描述的是 **架构与集成设计**，不是具体实现步骤；后续的实现计划会单独写在 implementation plan 中。

---

## 1. 目标（Goals）

- **在 App Router 中集成 next-intl**，支持 `en` 与 `zh` 两种语言  
- **引入顶层 `[locale]` 路由前缀**，在不破坏现有 API 与 SSG 行为的前提下，实现基于 URL 的多语言  
- **合并 next-intl middleware 与现有 API 中间件**，保证 API 鉴权、demo 模式、SSE 等逻辑完全不受 i18n 影响  
- **在 Header 中提供语言切换按钮**，整体风格与当前 UI 保持一致

---

## 2. 目录与路由结构（Directory & Routing Structure）

### 2.1 当前结构（Current State）

- App Router 根目录：`src/app`  
- 根布局：`src/app/layout.tsx`  
  - 负责渲染 `<html>` 与 `<body>`  
  - 引入字体与 `DemoBanner`
- 主要页面：
  - `src/app/page.tsx` —— 首页（home）  
  - `src/app/settings/page.tsx` —— 设置页（settings）  
  - `src/app/workspace/[slug]/page.tsx` —— 工作区主视图（main workspace）  
  - `src/app/workspace/[slug]/activity/page.tsx` —— 工作区活动页（workspace activity）

### 2.2 目标结构（Target Structure）

在 App Router 顶层引入动态段 `[locale]`，并将现有页面迁移到该段下：

```text
src
└── app
    ├── layout.tsx                  // 根布局（不再渲染 <html>）
    └── [locale]
        ├── layout.tsx              // 按 locale 区分的布局
        ├── page.tsx                // 首页（由原 app/page.tsx 迁移）
        ├── settings
        │   └── page.tsx            // 设置页（由原 app/settings/page.tsx 迁移）
        └── workspace
            └── [slug]
                ├── page.tsx        // 工作区主视图
                └── activity
                    └── page.tsx    // 工作区活动页
```

具体约定：

- 顶层 `layout.tsx`：
  - 只负责引入 `globals.css`  
  - 导出 `metadata`  
  - **不再**渲染 `<html>` 与 `<body>`

- `[locale]/layout.tsx`：
  - 校验 `params.locale` 是否在受支持的语言列表内  
  - 动态加载 `messages/{locale}.json`  
  - 渲染 `<html lang={locale}>` 与 `<body>`  
  - 在内部挂载 `NextIntlClientProvider`  
  - 保留原有字体和 `DemoBanner`

## 3. i18n 配置与文案字典（i18n Config & Message Dictionaries）

### 3.1 路由配置（Routing Configuration）

新增 `src/i18n/routing.ts`，用来统一 next-intl 与 App Router 的 locale 路由规则：

- 使用 `defineRouting` 定义：
  - `locales: ['en', 'zh']`  
  - `defaultLocale: 'en'`  
  - `localePrefix: 'always'` —— 所有页面 URL 显式带前缀，例如 `/en/...`、`/zh/...`
- 可选：使用 `createSharedPathnamesNavigation` 导出：
  - `Link`、`useRouter`、`usePathname` 等导航辅助函数
- Header 中的语言切换器会使用这些 helper 来进行路由切换。

### 3.2 文案字典（Message Dictionaries）

字典文件位置：

- `messages/en.json`  
- `messages/zh.json`

设计原则：

- key 使用语义化命名，例如：
  - `header.title`  
  - `header.allWorkspaces`  
  - `header.language.en` / `header.language.zh`
- JSON 仅存放纯文案，不掺入业务逻辑  
- 中英文注释放在使用这些 key 的 TS/TSX 文件（如 `Header`、i18n 配置等），不写入 JSON 里

示例 key：

- `header.title` —— 应用标题  
- `header.allWorkspaces` —— “返回所有工作区”按钮文案  
- `header.online` / `header.offline` —— 在线 / 离线状态文案  
- `header.language.en` / `header.language.zh` —— 语言选项展示名称

## 4. middleware 合并设计（Merged Middleware Design）

### 4.1 当前 middleware 情况（Current State）

现有的 `src/middleware.ts`：

- `config.matcher: '/api/:path*'`，只作用于 `/api` 路由  
- 主要职责：
  - `MC_API_TOKEN` 鉴权  
  - `DEMO_MODE` 只读模式控制  
  - `/api/events/stream` 的 token 特例处理（例如 query token）

### 4.2 合并后的结构（Target Structure）

在保证 API 行为不变的前提下，将 next-intl middleware 与现有 middleware 合并。

逻辑分层：

1. **next-intl middleware 初始化**

   ```ts
   const handleI18n = createMiddleware(routing);
   ```

2. **保留原有辅助函数与常量**

   如：`MC_API_TOKEN`、`DEMO_MODE`、`isSameOriginRequest` 等，原样搬运。

3. **入口函数 `middleware(request)` 拆分为两个分支**

伪代码结构：

```ts
export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // 1) API 分支：完全跳过 next-intl
  if (pathname.startsWith('/api/')) {
    // 这里包裹原有 API middleware 逻辑（只增加一层缩进，不改变语义）：
    // - DEMO_MODE 写操作拦截
    // - MC_API_TOKEN 鉴权
    // - /api/events/stream 的 query token 特例
    // - 默认 NextResponse.next()
  }

  // 2) 页面分支：交给 next-intl 处理
  return handleI18n(request);
}
```

4. **matcher 扩展配置**

```ts
export const config = {
  matcher: [
    // 页面路由：匹配所有非 API / 非 _next / 非 _vercel，且不包含点号的路径
    '/((?!api|_next|_vercel|.*\\..*).*)',

    // API 路由：保留对 /api/:path* 的显式匹配
    '/api/:path*'
  ]
};
```

**关键保证：**

- 所有 `/api/*` 请求只经过“原有 API 逻辑”，**不会**被 next-intl 改写或拦截  
- 所有非 `/api` 请求由 next-intl 处理 locale 识别与 URL 重写

## 5. Header 语言切换器设计（Header Language Switcher）

### 5.1 目标（Goals）

- 在 `src/components/Header.tsx` 中增加语言切换 UI  
- 样式与现有按钮（如 Settings 按钮、顶部状态 chips）保持一致  
- 切换语言时，尽量只改变 URL 前缀（`/en` ↔ `/zh`），其余路径结构保持不变

### 5.2 技术实现（Technical Implementation）

- 从 `src/i18n/routing.ts` 导入 `useRouter` 与 `usePathname`（或直接从 `next-intl/navigation` 导入）  
- 使用 `usePathname()` 获取当前路径，如 `/en/workspace/demo/activity`

路径解析示例：

```ts
const pathname = usePathname();
const segments = pathname.split('/');
const currentLocale = segments[1] || 'en';
const restPath = '/' + segments.slice(2).join('/');
```

当用户选择目标语言 `targetLocale` 时：

```ts
const nextPath = `/${targetLocale}${restPath}`;
router.push(nextPath);
```

**UI 形态：**

- 桌面端（Desktop）：
  - 在 Header 右侧区域（时钟、ONLINE 状态、Settings 按钮附近）增加一个语言切换按钮组，例如 `EN` / `中文`
- 移动端（Mobile / Portrait）：
  - 在纵向布局的 header 中，紧挨 Settings 图标，使用相同尺寸的圆角按钮或简洁下拉
- 所有新增 JSX 与逻辑行建议附带中英对照注释，方便后续维护

## 6. 合规性与风险控制（Compliance & Risk Control）

- **API 行为保持不变：**
  - API 请求只匹配 `/api/:path*`，并仅执行原有逻辑  
  - next-intl 的 middleware matcher 显式排除了 `api` 前缀

- **SSG / 路由稳定性：**
  - 页面迁移过程中，只改变物理路径与 URL 前缀，内部数据获取与渲染逻辑保持不变  
  - 将来如果某些页面需要 `generateStaticParams` 等 SSG 特性，可以基于 `[locale]` 结构安全扩展

- **防御式实现：**
  - 在 middleware 中使用 `pathname.startsWith('/api/')` 明确区分 API 和 页面分支  
  - 不对现有 API 逻辑做重构，只做“包裹 + 缩进”层面的调整  
  - 如果在实现过程中发现 locale 前缀与现有路径发生冲突（如静态资源访问异常），应立即停止调整并在文档中记录问题

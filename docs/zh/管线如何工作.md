# 管线如何工作

Mission Control 使用一个多智能体管线，将任务从想法自动推进到完成。每个阶段都有专门的智能体负责，系统会在各阶段之间自动完成交接。

---

## 管线阶段

当你创建一个任务并启动它时，会发生下面这些事情：

### 1. 📋 规划（Planning）
你描述你想要构建的东西。系统会将需求拆解成子任务，分配给合适的智能体，并把任务放入队列。

### 2. 🔨 构建（Building，进行中）
**Builder** 智能体领取任务并开始写代码。它会阅读任务说明、相关文件，以及——非常关键的——从以往任务中沉淀下来的经验。当它完成后，会登记自己的交付物（文件、URL），并把任务标记为“准备测试”。

### 3. 🧪 测试（Testing）
**Tester** 智能体会对每一个交付物运行自动化检查。它会启动真实浏览器（Playwright），并验证以下四类问题：

| 检查项 | 能发现什么问题 |
|--------|----------------|
| **JavaScript 错误** | 控制台错误、未捕获异常 |
| **CSS 校验** | 样式表语法错误 |
| **资源加载** | 图片 404、脚本丢失、静态资源加载失败 |
| **HTTP 状态码** | URL 交付物的服务器错误 |

Tester 还会生成完整页面截图，作为测试证据。

- ✅ **所有检查通过** → 任务进入 Review 阶段
- ❌ **任一检查失败** → 带着具体错误信息退回给 Builder

### 4. 👀 代码评审（Review）
**Reviewer** 智能体会审查代码本身——不仅看它“能不能跑”，还会看它是不是 **写得好**。它关注：

- 代码质量与可读性
- 架构与设计决策
- 安全性问题
- 实现是否符合最初的任务说明

可以把 Testing 理解为“它究竟能不能用？”，而 Review 则是“它是不是一个好的实现？”。

- ✅ **评审通过** → 任务进入 Verification 阶段
- ❌ **评审失败** → 携带评审意见退回给 Builder

### 5. ✅ 验证（Verification）
**Verifier** 智能体负责最终验收。这是进入“Done”之前的最后一道关。它要确认交付物 **真正满足** 原始任务需求——不仅代码能跑、看起来不错，而且 **确实做到了用户一开始要求的事情**。

- ✅ **验证通过** → 任务进入 **Done** 🎉
- ❌ **验证失败** → 退回给 Builder

---

## 循环回退机制（Loop-Back）

核心设计原则是：**任何环节失败，任务都会带着清晰的失败原因回到 Builder 手中。**

Builder 并不是从零开始重做，而是拿到：

- 精确的报错信息
- 测试截图
- 评审备注
- 验证阶段的反馈

这样每一轮都是“有针对性的改进”，而不是“盲目重试”。

```
Builder → Tester（失败："3 个图片链接失效"）
    ↑          |
    └──────────┘  Builder 只修这 3 个图片链接

Builder → Tester（通过）→ Reviewer（失败："搜索存在 SQL 注入风险"）
    ↑                             |
    └─────────────────────────────┘  Builder 修复 SQL 问题

Builder → Tester（通过）→ Reviewer（通过）→ Verifier（通过）→ Done ✅
```

系统会记录重试次数，并使用指数退避（10 秒 → 20 秒 → 40 秒，直到 5 分钟）。如果连续 5 次都失败，任务会被标记为“需要人工介入”，避免无限循环。

---

## Learner（学习器）

**Learner** 并不是一个独立阶段，而是挂在 **每一次阶段切换** 上的一个钩子。每当任务在任意阶段通过或失败时，Learner 都会收到通知，并记录：

- **哪里出错了**（如果失败）
- **是怎样的模式导致了这个问题**
- **下次如何提前避免**
- **未来任务可复用的检查清单项**

这些经验会写入知识库。下一次 Builder 开始处理新任务时，最相关的经验会直接注入到它的上下文里。比如，上周因为 CSS import 问题导致过失败，那么这次 Builder 在写任何 CSS 相关代码前，就已经知道要避开这类坑。

**每一次失败，都会让之后的每一次构建变得更聪明。**

---

## 并发与排队

系统会限制同时“在管线中处理”的任务数量（默认：3 个）。如果管线已经满了，新任务会进入等待队列，并在有空位时自动继续推进。

这样可以防止智能体之间互相“挤爆”，让整条管线稳定、平滑地流动。

---

## 工作流配置

这条管线是完全可配置的，你可以：

- **自定义阶段** —— 增加、删除或调整阶段顺序
- **配置失败回流目标** —— 每个阶段失败后要退回到哪里（不一定必须回 Builder）
- **为不同任务分配不同的智能体角色**
- **通过工作区里的 `WORKFLOW.md` 或数据库进行配置**

默认工作流是：`Builder → Tester → Reviewer → Verifier → Done`，对大多数项目已经足够。但你也可以为不同类型的工作创建特定模板。

## 快速参考

| 智能体 | 角色 | 职责 | 通过时 | 失败时 |
|--------|------|------|--------|--------|
| **Builder** | 构建 | 编写代码 | → Testing | — |
| **Tester** | 测试 | 自动化浏览器检查 | → Review | → Builder |
| **Reviewer** | 评审 | 代码质量与架构审查 | → Verification | → Builder |
| **Verifier** | 验证 | 需求验收 | → Done | → Builder |
| **Learner** | 学习 | 记录与提炼经验 | *(始终运行)* | *(始终运行)* |

---

## 总结

1. 你描述你想要的结果  
2. Builder 负责编写实现  
3. Tester 确认它“能正常工作”  
4. Reviewer 检查它“是不是一个好实现”  
5. Verifier 核对它“是不是真正满足最初的需求”  
6. Learner 在每一步都记录经验  
7. 任一步失败都会带着明确原因回到 Builder  
8. 每一次迭代都会在上一轮经验的基础上变得更好

最终目标是：**你只需要说出你想要什么，管线会自动循环迭代，直到“真正做到对”为止，而不用你亲自盯住每一个细节步骤。**

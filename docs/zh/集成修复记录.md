# Mission Control 集成修复记录（Integration Fixes）

**日期：** 2025-01-31

## 已修复问题（Problems Fixed）

### 1. ✅ 实时更新无效（Real-time Updates Not Working）

**问题：** 任务状态变更后需要手动刷新页面才能看到更新  
**根因：** 派发（dispatch）接口没有向 SSE 管道广播事件  
**修复：** 在 `/api/tasks/[id]/dispatch` 路由中，在更新任务状态之后增加 `broadcast()` 调用。

**变更文件：**

- `src/app/api/tasks/[id]/dispatch/route.ts`

**测试结果：** 任务状态变更后可以在 UI 中自动更新，无需刷新。

---

### 2. ✅ 智能体计数始终为 0（Agent Counter Shows 0）

**问题：** 即使子智能体正在工作，侧边栏仍显示 “0 agents”  
**根因：** Header 和侧边栏没有查询 `openclaw_sessions` 中“活跃子智能体”的数量  
**修复：**

- 在 `AgentsSidebar` 中补全对子智能体数量的查询（原结构已存在）  
- 在 `Header` 组件中增加对子智能体数量的查询  
- 头部统计中将“正在工作的主智能体数量 + 活跃子智能体数量”相加展示

**变更文件：**

- `src/components/Header.tsx`

**测试结果：** 当有子智能体在工作时，计数会正确显示 “1 agent active”。

---

### 3. ✅ 活动 / 交付物 / 会话为空（Activities / Deliverables / Sessions Empty）

**问题：** 无法从 UI 中看出子智能体正在做什么  
**根因：** orchestrator 的编排逻辑没有向活动和交付物相关的 API 写入数据  
**修复：** 创建一个完备的编排辅助库，用于封装所有日志写入。

**新增文件：**

- `src/lib/orchestration.ts` —— 所有编排相关的辅助函数  
- `docs/CHARLIE_WORKFLOW.md` —— orchestrator 的完整使用指南

**API 辅助函数：**

- `logActivity()` —— 记录任务活动  
- `logDeliverable()` —— 记录创建的文件 / URL 等交付物  
- `registerSubAgentSession()` —— 在 DB 中注册子智能体会话  
- `completeSubAgentSession()` —— 标记会话为完成  
- `onSubAgentSpawned()` —— 封装“子智能体启动”全流程  
- `onSubAgentCompleted()` —— 封装“子智能体完成”全流程  
- `verifyTaskHasDeliverables()` —— 在批准前检查是否有交付物

**测试结果：** 当 orchestrator 使用这些辅助函数时，Activity / Deliverables / Sessions 三个标签页都会显示数据。

---

### 4. ✅ 缺失 Review 工作流（Review Workflow Missing）

**问题：** 任务自动流转到 REVIEW 后可以直接被标记为 DONE，中间没有任何校验  
**根因：** 在把任务状态从 REVIEW 改为 DONE 时，没有验证是否存在交付物  
**修复：** 在任务 `PATCH` 接口中加入“交付物检查”逻辑。

**变更文件：**

- `src/app/api/tasks/[id]/route.ts`

**校验逻辑示例：**

```typescript
// 在允许 review -> done 状态变更之前：
const deliverables = queryAll<TaskDeliverable>(
  'SELECT * FROM task_deliverables WHERE task_id = ?',
  [id]
);

if (deliverables.length === 0) {
  return NextResponse.json(
    { error: 'Cannot approve task: no deliverables found...' },
    { status: 400 }
  );
}
```

**测试结果：** 如果任务没有任何交付物，API 会拒绝从 REVIEW 流转到 DONE。

---

### 5. ✅ Header 统计数据不准确（Header Stats Wrong）

**问题：** Header 中显示的 “0 agents active, 4 tasks” 与实际情况不符  
**根因：**

- 智能体数量没有包含子智能体  
- 任务数量统计中包括了已完成 / 已评审的任务

**修复：**

- 智能体数量：使用 `workingAgents + activeSubAgents`  
- 任务数量：只统计 `inbox`, `assigned`, `in_progress` 三种状态，排除 `review` 和 `done`

**变更文件：**

- `src/components/Header.tsx`

**测试结果：** Header 中的智能体与任务数量统计与实际状态一致。

## orchestrator 使用示例（Usage for the orchestrator）

在 orchestrator 为 Mission Control 任务创建子智能体时，典型用法如下：

```typescript
import * as orchestrator from '@/lib/orchestration';

// 1. 子智能体刚刚启动时
await orchestrator.onSubAgentSpawned({
  taskId: 'task-id',
  sessionId: 'agent:main:subagent:xyz',
  agentName: 'my-subagent',
  description: 'Task description'
});

// 2. 工作过程中记录进度
await orchestrator.logActivity({
  taskId: 'task-id',
  activityType: 'updated',
  message: 'Fixed something'
});

// 3. 完成时记录总结与交付物
await orchestrator.onSubAgentCompleted({
  taskId: 'task-id',
  sessionId: 'agent:main:subagent:xyz',
  agentName: 'my-subagent',
  summary: 'Completed successfully',
  deliverables: [
    { type: 'file', title: 'My file', path: 'src/...' }
  ]
});

// 4. 在批准任务前检查是否有交付物
const ok = await orchestrator.verifyTaskHasDeliverables('task-id');
if (ok) {
  // Approve task
}
```

完整细节请参考 `docs/CHARLIE_WORKFLOW.md`。
